Iceberg Compaction & Maintenance Operational Guide

Customer: State Bank of India (SBI)CDP Private Cloud Base 7.3.1.600+Spark 3.5.4 · Iceberg 1.5.2 · HDFS / Ozone

This is the operational guide for maintaining data files, delete files, snapshots, metadata, and orphan files on SBI's PROD Iceberg tables. It has been reviewed and revised against the Iceberg 1.5.2 Spark Procedures reference and known CDP operational risks.

Must-verify before approval (P0)
Every CALL statement in this guide requires Spark 3's Iceberg SQL extension and a correctly configured Iceberg catalog — without these two, every procedure fails, regardless of authorization setup. Before production rollout, use a maintenance-only Spark job to confirm the extension/catalog setup and read/write/delete access to the underlying HDFS/Ozone paths. If your deployment enforces authorization through Ranger, also confirm the required Ranger policies are in place; Ranger itself is not a prerequisite for these procedures on deployments that don't route through it.

1. Scope and Common Operating Principles

  • Each maintenance run is scoped to one table at a time. Never overlap it with ETL, MERGE, DELETE, UPDATE, or streaming commits on the same table.
  • Downtime is normally not required, but maintenance generates heavy read/shuffle/write activity and Iceberg commits. Use a dedicated queue and cap executors, memory, and concurrent runs.
  • Before and after each run, record the table's file/snapshot/manifest counts, Spark execution time, scan task skew, and query planning time.
  • Replace the example spark_catalog and databases.table with your actual catalog name and fully-qualified table name.
  • For rewrite_data_files, scope each run to partitions whose ingestion and read activity has already finished, using a where predicate (e.g. a date or partition-key filter) instead of rewriting the whole table.
  • rewrite_position_delete_files, rewrite_manifests, expire_snapshots, and remove_orphan_files operate at table scope in Iceberg 1.5.2, not partition scope — rewrite_data_files is the only procedure in this guide with a documented where predicate. Treat these four as a table-level maintenance window rather than something you can lock to a single partition.
Execution environment scope: This guide is written against Spark on YARN. OpenShift is not a prerequisite for the Iceberg procedure syntax or for the HDFS/Ozone file lifecycle described here. If maintenance Spark jobs instead run through CDE or Kubernetes executors, translate every reference to "YARN queue / executor / memory cap" in this document into that platform's equivalent controls — virtual cluster quota, pod CPU/memory limits, and concurrent job caps. Do not mix the resource-control model of the two execution paths within the same job.

2. Recommended Execution Order

Pre-checks & table lock→rewrite_data_files→rewrite_position_delete_files (conditional)→rewrite_manifests (conditional)→expire_snapshots
Orphan file cleanup is not forced into this chain. Run it as a separate weekly job only after a failed write or during a scheduled review, and always require a dry_run review and sign-off before deleting anything.

3. Data File Compaction: rewrite_data_files

Applicable 512 MB is a workload-specific starting point for benchmarking, not a fixed officially recommended value — validate it against the table's actual query and write pattern on both HDFS and Ozone. A target file size is not the same concept as an HDFS block size, and neither 128 MB nor Ozone's 5 GB should be treated as a fixed standard.

CALL spark_catalog.system.rewrite_data_files(
  table => 'databases.table',
  where => 'partition_col = ''2026-07-20''',
  strategy => 'binpack',
  options => map(
    'target-file-size-bytes', '536870912',
    'min-input-files', '5',
    'max-concurrent-file-group-rewrites', '2',
    'max-file-group-size-bytes', '21474836480',
    'partial-progress.enabled', 'false'
  )
);
  • The where predicate scopes the run to partitions whose ingestion and read window has already closed. This is the only one of the five procedures in this guide that accepts partition-level filtering — replace partition_col and the example value with your table's real partition key.
  • min-input-files=5 is one of several criteria used to decide which file groups get rewritten — it does not mean file groups with fewer than 5 files are always excluded.
  • The default for partial-progress.enabled is false. Setting it to true splits one job into multiple commits/snapshots, so only enable it when you're willing to accept more snapshots and a higher chance of colliding with concurrent writes. It is not a "required setting for large tables."
  • 512 MB is a target, not a guarantee. Actual file sizes may come out smaller or larger depending on partition boundaries, Spark task size, and compression ratio.
ALTER TABLE databases.table SET TBLPROPERTIES (
  'write.target-file-size-bytes' = '536870912'
);

Compaction options alone do not stop future writes from creating small files again. Set the write-side property above as well, and separately benchmark your Spark write-partitioning and AQE settings.

4. Position Delete Cleanup: rewrite_position_delete_files

Conditional This procedure rewrites position delete files (not equality deletes). It has two distinct uses: ordinary compaction of small position delete files, and removal of dangling deletes after rewrite_data_files.

For TSB 2024-752 mitigation, run it immediately after rewrite_data_files in the same maintenance window when the table is format v2, position delete files exist, and a downstream plain SELECT count(*) can be answered by Impala from statistics. This risk applies when the relevant delete files are fully dangling — they only reference data files removed by the preceding rewrite_data_files.

-- TSB 2024-752 mitigation: rewrite every position delete file
CALL spark_catalog.system.rewrite_position_delete_files(
  table => 'databases.table',
  options => map(
    'rewrite-all', 'true',
    'max-concurrent-file-group-rewrites', '2',
    'max-file-group-size-bytes', '21474836480'
  )
);

For ordinary position-delete compaction where complete dangling-delete removal is not required, use size-based selection instead:

CALL spark_catalog.system.rewrite_position_delete_files(
  table => 'databases.table',
  options => map(
    'min-input-files', '2',
    'max-concurrent-file-group-rewrites', '2',
    'max-file-group-size-bytes', '21474836480'
  )
);
Note: This procedure operates at table scope and has no documented where predicate — unlike rewrite_data_files, you cannot restrict it to specific partitions. Do not run the full rewrite-all TSB mitigation unconditionally on every table Impala queries; it increases scan and rewrite cost.

5. Manifest Rewrite: rewrite_manifests

Conditional This procedure does not delete old metadata JSON files. It rewrites and clusters manifest entries by partition fields to optimize scan planning, so use it only on tables with frequent partition filtering and a large number of manifests.

CALL spark_catalog.system.rewrite_manifests(
  table => 'databases.table',
  use_caching => false
);

Start large manifest jobs with use_caching=false to avoid executor memory pressure. If planning time doesn't improve before and after the run, drop the table from your regular schedule. This runs at table scope: in Iceberg 1.5.2, spec_id can select a partition specification, but it cannot filter to a specific partition value. Treat it as a table-level task, not a per-partition one.

6. Snapshot Expiry: expire_snapshots

Requires a retention policy The 30-day / 20-snapshot values below are only a starting example — set the real numbers based on your time-travel, audit, incident-analysis, and rollback requirements.

CALL spark_catalog.system.expire_snapshots(
  table => 'databases.table',
  older_than => TIMESTAMPADD(DAY, -30, CURRENT_TIMESTAMP),
  retain_last => 20,
  max_concurrent_deletes => 4
);

This job never deletes files referenced by a live snapshot, but once a snapshot expires you can no longer time-travel or roll back to it. Tune max_concurrent_deletes based on the load it places on HDFS/Ozone and the NameNode/OM. Snapshot history is inherently table/branch-scoped — there is no partition-level equivalent, so this runs against the whole table every time.

7. Metadata JSON Retention

expire_snapshots alone does not fully manage the lifecycle of old metadata JSON files. Only write.metadata.delete-after-commit.enabled=true is strictly required to start automatic deletion — with it enabled, old metadata files beyond the default retention count (100) are already deleted after each commit even if you never touch the second property below.

ALTER TABLE databases.table SET TBLPROPERTIES (
  'write.metadata.delete-after-commit.enabled' = 'true',
  'write.metadata.previous-versions-max' = '100'
);
Watch your retention count: write.metadata.previous-versions-max is not mandatory on its own, but setting it explicitly is recommended so you control the retention count instead of relying on the default. For a streaming table, 100 commits can pass in a very short time — decide the metadata JSON retention count per table based on commit frequency and how far back you need to be able to investigate an incident.

8. Orphan File Cleanup: remove_orphan_files

Weekly, approval-gated task This handles physical files left behind by failed Spark jobs, YARN container kills (or pod evictions / OOMKilled events if running on CDE or Kubernetes executors), or network failures that were never referenced by the table metadata. Listing the entire table location can put significant load on HDFS/Ozone if the table is large.

-- Step 1: review delete candidates
CALL spark_catalog.system.remove_orphan_files(
  table => 'databases.table',
  older_than => TIMESTAMPADD(DAY, -7, CURRENT_TIMESTAMP),
  dry_run => true
);

-- Step 2: after review and approval, delete for real
CALL spark_catalog.system.remove_orphan_files(
  table => 'databases.table',
  older_than => TIMESTAMPADD(DAY, -7, CURRENT_TIMESTAMP)
);

The retention window must be longer than the longest-running write you expect. 7 days is a conservative starting point — set the real value based on your actual longest write/retry duration. This scan walks the entire table location; there is no partition-level equivalent, so treat it as a table-level task like the three procedures above.

9. File Size and Scheduling Standards

10. Deployment Approval Checklist

  1. Verified the Iceberg SQL extension and catalog from a maintenance Spark job beforehand, plus storage permissions and, where Ranger is enabled, the required Ranger policies.
  2. Checked the target table's format version, delete-file type, and the scale of its snapshots/files/manifests.
  3. Implemented per-table locking and a schedule that never overlaps with ETL/streaming; for rewrite_data_files, confirmed the target partitions have finished ingestion and are not being actively read before the run.
  4. Set executor, memory, and concurrent-run caps on a dedicated resource pool — a YARN queue, or the equivalent CDE virtual cluster / Kubernetes namespace quota if maintenance runs on CDE or Kubernetes executors.
  5. Ran the first execution on a representative table and measured file count, scan performance, planning time, skew, and commit failure rate.
  6. Orphan cleanup only runs after the dry-run results are reviewed and explicitly approved.
