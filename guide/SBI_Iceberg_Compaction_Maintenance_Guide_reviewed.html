Iceberg Compaction & Maintenance Operational Guide

Customer: State Bank of India (SBI)CDP Private Cloud Base 7.3.1.600+Spark 3.5.4 · Iceberg 1.5.2 · HDFS / Ozone

SBI PROD Iceberg 테이블의 data file, delete file, snapshot, metadata 및 orphan file 유지보수를 위한 운영 가이드입니다. Iceberg 1.5.2 Spark Procedures와 CDP 운영 위험을 기준으로 검토·수정했습니다.

승인 전 필수 확인(P0)
이 문서의 CALL 구문은 Spark 3의 Iceberg SQL extension과 정상적인 Iceberg catalog 설정이 있어야 실행됩니다. production 적용 전 maintenance 전용 Spark Job에서 catalog, extension, Ranger 권한 및 HDFS/Ozone 경로의 read/write/delete 권한을 검증하십시오. 설정이 없으면 모든 procedure가 실패합니다.

1. 적용 범위와 공통 운영 원칙

  • 작업 단위는 테이블별 단일 maintenance run입니다. 동일 테이블에서 ETL, MERGE, DELETE, UPDATE, streaming commit과 중첩하지 않습니다.
  • 파티션 단위 compaction 정책: 파티션 테이블에서는 maintenance 대상 파티션에 ingest/write가 진행 중이거나, 해당 파티션을 읽는 배치·조회가 실행 중일 때 rewrite_data_files를 시작하지 않습니다. 대상 파티션의 write 및 read window가 모두 끝난 뒤 실행합니다.
  • Iceberg snapshot isolation상 동시 read 자체가 데이터 정합성을 깨뜨리지는 않지만, 기존 snapshot을 읽는 query와 compaction은 HDFS/Ozone·YARN 자원 경합 및 파일 수명 관리를 복잡하게 만듭니다. 따라서 위 read 비중첩은 운영 안정성과 예측 가능한 성능을 위한 필수 정책입니다.
  • 서비스 중단은 보통 필요 없지만 maintenance는 대규모 read/shuffle/write와 Iceberg commit을 발생시킵니다. 전용 YARN queue와 executor·memory·동시 실행 상한을 사용합니다.
  • 각 작업 전후 metadata table의 files/snapshots/manifests 수, Spark 실행시간, scan task skew 및 query planning 시간을 기록합니다.
  • 예시의 spark_catalog과 databases.table은 실제 catalog와 완전 수식 테이블명으로 대체합니다.
실행 환경 범위: 본 가이드는 Spark on YARN을 기준으로 작성되며, OpenShift는 Iceberg procedure의 문법이나 HDFS/Ozone 파일 lifecycle의 전제조건이 아닙니다. 만약 maintenance Spark Job을 CDE 또는 Kubernetes executor로 실행한다면, 이 문서의 “YARN queue / executor / memory 상한”은 해당 플랫폼의 virtual cluster quota, pod CPU·memory limit 및 동시 job 제한으로 치환하십시오. 두 실행 모델의 리소스 제어를 혼용하지 않습니다.

2. 권장 실행 순서

사전 점검·table lock→rewrite_data_files→rewrite_position_delete_files (조건부)→rewrite_manifests (조건부)→expire_snapshots
파티션 가용성 확인: rewrite_data_files 전에는 대상 파티션의 ingest/write와 read job이 없는지 확인하고 lock을 확보합니다. 파티션별로 순차 실행하며, 서로 다른 파티션도 YARN/Ozone 용량 여유가 검증되기 전에는 병렬 실행하지 않습니다.
Orphan file 정리는 위 체인에 강제 편입하지 않습니다. 실패한 write가 있거나 정기 점검이 필요한 경우에만 별도의 주간 작업으로 실행하며, 매번 dry_run 검토와 승인 단계를 거칩니다.

3. 데이터 파일 컴팩션: rewrite_data_files

적용 가능 512 MB는 HDFS와 Ozone 모두에서 합리적인 시작값입니다. 파일 목표 크기는 HDFS block size와 같은 개념이 아니며, 128 MB 또는 Ozone 5 GB를 고정 표준으로 사용하지 않습니다.

CALL spark_catalog.system.rewrite_data_files(
  table => 'databases.table',
  strategy => 'binpack',
  where => 'business_date = DATE ''2026-07-21''',
  options => map(
    'target-file-size-bytes', '536870912',
    'min-input-files', '5',
    'max-concurrent-file-group-rewrites', '2',
    'max-file-group-size-bytes', '21474836480',
    'partial-progress.enabled', 'false'
  )
);
  • where는 대상 파티션을 지정하는 예시입니다. 실제 partition transform과 컬럼 타입에 맞는 predicate로 교체합니다. 이 가이드는 파티션 테이블에서 이 방식의 partition-at-a-time 실행을 기본으로 합니다.
  • min-input-files=5는 5개 이상 file group을 재작성 대상으로 삼는 하나의 기준입니다. 5개 미만 file group이 항상 제외된다는 뜻은 아닙니다.
  • partial-progress.enabled의 기본값은 false입니다. true는 하나의 작업을 여러 commit/snapshot으로 나누므로 snapshot 증가와 외부 write 충돌 가능성을 감수할 때만 사용합니다. “대형 테이블 필수” 옵션이 아닙니다.
  • 512 MB는 목표치이지 보장값이 아닙니다. 파티션 경계, Spark task 크기 및 압축률에 따라 실제 파일은 더 작거나 클 수 있습니다.
ALTER TABLE databases.table SET TBLPROPERTIES (
  'write.target-file-size-bytes' = '536870912'
);

Compaction 옵션만으로는 이후 write의 small-file 생성을 막지 못합니다. 신규 write 정책도 함께 설정하고, Spark write partition/AQE 설정을 별도로 실측 튜닝하십시오.

4. Position delete 정리: rewrite_position_delete_files

조건부 적용 이 절차는 equality delete가 아닌 position delete를 재작성하고 dangling delete를 제거합니다. 다음 조건이 모두 해당될 때 rewrite_data_files 직후 같은 maintenance window에서 실행합니다.

  • Iceberg format v2 테이블이며 position delete 파일이 존재한다.
  • Spark rewrite_data_files를 수행했다.
  • Impala의 stats-optimized plain SELECT count(*) 정확성 위험을 제거해야 한다.
CALL spark_catalog.system.rewrite_position_delete_files(
  table => 'databases.table',
  options => map(
    'min-input-files', '2',
    'max-concurrent-file-group-rewrites', '2',
    'max-file-group-size-bytes', '21474836480'
  )
);
주의: 모든 Impala 조회 테이블에 무조건 실행할 작업은 아닙니다. 조건에 맞지 않는 테이블에서 전면 실행하면 불필요한 scan과 commit만 증가합니다.
범위 제약: Iceberg 1.5.2의 rewrite_position_delete_files procedure는 where 인자를 제공하지 않습니다. 따라서 이 절차는 partition-at-a-time 작업으로 가정할 수 없으며 table scope로 계획해야 합니다. 실행 시에는 해당 테이블의 write 및 중요 read window 전체와 겹치지 않게 스케줄하십시오.

5. Manifest 재작성: rewrite_manifests

조건부 적용 이 절차는 오래된 metadata JSON을 삭제하는 작업이 아닙니다. manifest 배치를 정리해 planning을 개선하며, 파티션 필터가 빈번하고 manifest 수가 많은 테이블에만 사용합니다.

CALL spark_catalog.system.rewrite_manifests(
  table => 'databases.table',
  use_caching => false
);

대형 manifest 작업에서는 executor 메모리 압박을 피하기 위해 use_caching=false부터 시작합니다. 실행 전후 planning time이 개선되지 않으면 정기 실행 대상에서 제외하십시오.

범위: rewrite_manifests도 table scope입니다. 파티션별 compaction이 끝난 뒤, 필요한 테이블에 대해 별도 maintenance window에서 실행합니다.

6. Snapshot 만료: expire_snapshots

업무 보존정책 필요 30일/20개는 시작 예시일 뿐이며, time travel, 감사, 장애 분석 및 rollback 요구사항에 따라 확정해야 합니다.

CALL spark_catalog.system.expire_snapshots(
  table => 'databases.table',
  older_than => TIMESTAMPADD(DAY, -30, CURRENT_TIMESTAMP),
  retain_last => 20,
  max_concurrent_deletes => 4
);

이 작업은 live snapshot이 참조하는 파일을 삭제하지 않지만, 만료된 snapshot으로 time travel 또는 rollback하는 것은 불가능해집니다. max_concurrent_deletes는 HDFS/Ozone 및 NameNode/OM 부하를 보며 조정합니다.

범위: snapshot 만료는 table scope입니다. 특정 파티션만 대상으로 만료할 수 없으므로 ingestion 및 중요 read window와 분리합니다.

7. Metadata JSON 보존

expire_snapshots만으로는 이전 metadata JSON lifecycle을 충분히 관리하지 못합니다. 아래 두 속성은 함께 설정해야 하며, 두 번째 속성만 설정해도 자동 삭제는 시작되지 않습니다.

ALTER TABLE databases.table SET TBLPROPERTIES (
  'write.metadata.delete-after-commit.enabled' = 'true',
  'write.metadata.previous-versions-max' = '100'
);
보존값 주의: streaming table에서 100 commits는 짧은 시간일 수 있습니다. metadata JSON 보존 수는 commit 빈도와 장애 조사에 필요한 기간을 함께 계산하여 테이블별로 정합니다.

8. Orphan 파일 정리: remove_orphan_files

주간·승인형 작업 실패한 Spark 작업, YARN container kill, 네트워크 장애로 남은 미참조 물리 파일을 처리합니다. table location 전체 listing이 크면 HDFS/Ozone 부하가 상당할 수 있습니다.

-- 1단계: 삭제 후보 확인
CALL spark_catalog.system.remove_orphan_files(
  table => 'databases.table',
  older_than => TIMESTAMPADD(DAY, -7, CURRENT_TIMESTAMP),
  dry_run => true
);

-- 2단계: 후보 검토·승인 후 실제 삭제
CALL spark_catalog.system.remove_orphan_files(
  table => 'databases.table',
  older_than => TIMESTAMPADD(DAY, -7, CURRENT_TIMESTAMP)
);

보존기간은 가장 오래 걸릴 수 있는 write의 최대 시간보다 길어야 합니다. 7일은 보수적인 시작값이며, 실제 장기 write/재시도 시간을 기준으로 정합니다.

범위: orphan 파일 정리도 table location을 대상으로 하므로 partition-at-a-time 작업이 아닙니다. table-level maintenance window와 승인 절차가 필요합니다.

9. 파일 크기 및 스케줄링 기준

항목운영 기준
HDFS / Ozone target file size512 MB에서 시작. table별로 256 MB~1 GB를 benchmark하여 확정. Ozone 5 GB를 기본값으로 사용하지 않음.
Compaction 주기파티션 단위로 순차 실행. 대상 파티션의 ingest/write 및 read가 없는 window만 사용하며, 전체 테이블 full rewrite를 정기 기본값으로 두지 않음.
Position deleteformat v2 + position delete + data-file rewrite 조건을 만족하는 테이블만 실행. Iceberg 1.5.2에서는 table scope이므로 table-level window 필요.
Snapshot expiry일 1회 가능. 업무 보존기간 및 최소 snapshot 수를 명시. table scope이므로 write/중요 read와 분리.
Orphan cleanup주 1회 또는 실패 이벤트 후. dry-run 검토 및 승인 후 실제 삭제. table scope으로 실행.
오케스트레이션Airflow, Oozie 또는 조직 표준 scheduler로 table lock, 순서, retry, log, alert를 관리. 특정 container platform은 본 문서의 전제조건이 아님.

10. 배포 승인 체크리스트

  1. maintenance Spark Job에서 Iceberg SQL extension, catalog, Ranger 및 storage 권한을 사전 검증했다.
  2. 대상 테이블의 format version, delete 유형, snapshots/files/manifests 규모를 확인했다.
  3. 대상 파티션의 ingest/write 및 read job이 모두 종료된 것을 확인하고 partition lock을 확보했다.
  4. table-scope procedure(position delete, manifest, snapshot, orphan)의 별도 maintenance window를 확보했다.
  5. 전용 YARN queue의 executor, memory, 동시 실행 상한을 설정했다.
  6. 첫 실행은 대표 파티션에서 시작해 파일 수, scan 성능, planning time, skew, commit 실패율을 측정했다.
  7. Orphan cleanup은 dry-run 결과를 검토하고 명시적으로 승인받은 뒤 실행한다.
